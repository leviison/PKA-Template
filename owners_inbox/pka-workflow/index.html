2b

Brief Lifecycle

A brief exists as a temporary markdown file in team_comms/ while work is in progress — the database row is the permanent record, with the file moved to archive/team_comms/ on completion (via archive.py, tracked in git). The Learning Layer adds a parallel post-delivery branch: the Learning Designer runs an AAR with the delivering team member and authors a case writeup in case_studies.

flowchart TD START(["Owner requests work"]) START --> LEROY_W["Leroy writes\nbrief_ref.md → Team Comms/"] LEROY_W --> DB_INSERT["INSERT INTO briefs\nstatus open"] DB_INSERT --> TEAM_READ["Team member reads brief file\nfrom filesystem — NOT from conversation\nContext isolation preserved"] TEAM_READ --> WORK["Team member executes work"] WORK --> DELIVER["Deliverable written to\nOwners Inbox/"] DELIVER --> DB_DELIV["INSERT INTO deliverables\nbriefs.id FK + file_path"] DB_DELIV --> FILE_MOVE["brief_ref.md MOVED to archive/team_comms/\nvia archive.py brief "] FILE_MOVE --> DB_UPDATE["UPDATE briefs SET\nstatus complete\ncompleted_at set"] DB_UPDATE --> FEEDBACK["Owner reviews deliverable\nand provides rating"] FEEDBACK --> DB_FEED["INSERT INTO feedback\nrating · notes · brief_ref"] DB_FEED --> DONE(["Work cycle complete"]) %% Learning Layer — parallel branch from delivery DB_DELIV --> AAR["Learning Designer runs AAR\n4 questions · post-delivery"] AAR --> CASE_WRITE["Case writeup drafted\n~800 words, HBS-style · authored by Learning Designer"] CASE_WRITE --> DB_CASE["INSERT INTO case_studies\nstatus published"] DB_CASE --> LEARN_DONE(["📚 Teaching artifact preserved"]) FILE_MOVE -.->|"FILE ARCHIVED\nrecoverable in git history"| NOTE_FILE(["📦 File preserved"]) DB_UPDATE -.->|"DB RECORD IS PERMANENT\nsurvives forever"| NOTE_DB(["🗄️ Row preserved"])
2c

Database Schema

The full pka.db entity-relationship diagram — assets feed content, briefs produce deliverables which collect feedback and AAR-driven case studies. Journal, knowledge, backlog, and settings are standalone. Model routing tables (model_providers, task_type_models) govern per-brief model tier assignment; team_members and feedback both carry FK references to model_providers. memory is the PKA-native durable memory layer with bi-temporal state and an FTS5 mirror, landed via migrations/001_memory_table.sql (pre-applied at install).
